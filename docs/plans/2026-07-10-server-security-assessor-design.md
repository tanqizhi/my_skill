# Server Security Assessor Skill 设计

日期：2026-07-10

## 1. 目标

创建一个模块化的单一 Codex skill，用于评估服务器和对外业务的安全状况。它支持三种评估模式：

1. 仅外部检查：主动扫描或针对既有漏扫报告进行验证，并可选择安全 PoC。
2. 仅内部检查：通过 SSH 登录主流 Linux 服务器执行内部安全检查。
3. Both：同时执行外部和内部检查，关联暴露端口、服务进程、软件包、配置与日志证据。

Windows 目标只支持外部检查。Linux 目标支持全部三种模式。第一版只评估并给出修复建议；设计为后续可扩展到经用户逐项确认后的自动修复。

## 2. 范围和基线

每次运行先选择模式，再选择检查范围：

- 全面检查：覆盖所选模式的完整检查清单。
- 定向检查：仅检查用户指定的 IP、域名、端口、URL、协议、服务、目录、配置、日志或时间范围。

用户提供目标即视为已获得授权，但 skill 不得自动发现或扩展到未提供的关联资产。若用户提供漏扫报告，报告中的目标和发现构成外部验证上限，即使选择全面检查也不得启动无关扫描。

默认安全基线由 CIS Level 1、厂商安全建议、CVE 风险和实际可利用性组成，并允许用户加载等保、行业规范或自定义检查清单。

## 3. 架构

采用一个 `server-security-assessor` skill，由 `SKILL.md` 统一负责输入、预检、模式路由、风险门禁、停止条件和结果交付。具体知识与确定性处理按模块拆分：

- `references/external-assessment.md`：主动扫描、安全 PoC 和 Web 检查规则。
- `references/report-validation.md`：已有报告的定向解析与验证。
- `references/linux-internal-assessment.md`：SSH 内部基线、漏洞和入侵迹象检查。
- `references/forensics.md`：独立或升级式深度取证。
- `references/baselines/`：默认基线、自定义基线接口和映射规则。
- `references/report-schema.md`：统一发现、证据、严重度和输出结构。
- `scripts/`：依赖检测、结果规范化、证据脱敏、去重和报告组装；脚本不自主作出安全结论。

主流程：

1. 预检评估机、目标和输入。
2. 选择 External、Internal 或 Both。
3. 选择全面或定向范围。
4. 执行选中的外部和/或内部流程。
5. 规范化、去重并关联证据。
6. 用户可独立启动取证；标准检查发现异常时也可展示触发证据并请求升级。
7. 生成技术报告、领导版 HTML、机器可读 JSON 和证据包。

## 4. 外部评估

无漏扫报告时，外部工具链为：

1. Nmap 发现端口、协议、服务和版本。
2. Nuclei 检查 CVE、错误配置、暴露文件和模板化问题，以 JSONL 输出供规范化。
3. 当目标被识别为 Web 业务时，允许追加长亭 Xray，进行爬虫和参数级 Web 安全检查。
4. 合并 Nuclei、Xray 和其他证据；同一问题不重复计分。
5. 根据用户选择，执行无破坏、无持久化、尽量不改变状态的安全 PoC。

若用户提供漏扫报告，只验证报告列出的目标和发现。Xray 只能针对报告中的 Web URL 或问题运行，不得启动全站爬取。

评估机支持 macOS 和主流 Linux。缺少 Nmap 或 Nuclei 时，按 Homebrew、APT、DNF/YUM 等环境安装。Xray 使用长亭官方的单文件二进制，必须区分于 XTLS/Xray-core；首次下载时展示许可证，选择匹配的架构并按官方 `sha256.txt` 校验。工具只安装在评估机，不自动安装到目标服务器。

参考：

- Nuclei 官方运行文档：https://docs.projectdiscovery.io/opensource/nuclei/running
- 长亭 Xray 快速开始：https://docs.xray.cool/tools/xray/QuickStart
- 长亭 Xray 扫描方式：https://docs.xray.cool/tools/xray/Scanning

## 5. Linux 内部评估

SSH 检查覆盖主流 Ubuntu/Debian 与 RHEL/CentOS/Rocky/Alma 系列。普通账户默认以当前权限执行，必要时经确认使用 `sudo`；若用户直接提供 root 账户，则以 root 执行只读检查。

全面检查包括：系统与补丁、账户与认证、SSH、安全策略、防火墙、监听服务、软件包、文件权限、内核参数、日志、计划任务、启动项、容器、可疑进程、异常网络连接、持久化机制及基础入侵痕迹。定向检查只访问用户指定的服务、目录、文件、日志或时间范围。

## 6. 深度取证

深度取证支持两种入口：

- 用户直接选择独立取证模式。
- 标准内部检查发现异常后，先展示触发证据，再请求用户确认升级。

取证覆盖进程、网络、登录、启动项、计划任务、异常文件、日志时间线和基础恶意软件调查。材料按敏感数据处理，仅采集与结论相关的最小内容，并记录采集时间、来源和 SHA-256。完整磁盘或内存镜像等高影响动作不属于默认流程，必须另行明确授权。

## 7. 统一发现和输出

所有证据进入统一证据层，记录采集时间、来源、原始文件、命令或请求摘要、退出状态和 SHA-256，并脱敏密码、密钥、Token、Cookie 及无关个人数据。

统一发现至少包含：发现编号、资产、受影响组件、来源、状态、严重度、可信度、实际暴露条件、证据引用、影响、修复建议、复测方法和基线映射。状态为：

- `confirmed`：证据足以确认。
- `probable`：高度可疑但证据不足。
- `not-reproduced`：本次未复现，并保留原因。
- `not-assessable`：因安全边界、权限、工具或环境限制无法检查。

输出包括：

- `report.md`：面向安全与运维人员的完整技术报告。
- `executive-report.html`：面向领导的自包含、响应式、可打印 HTML，展示总体评分、风险分布、关键暴露面、Top 风险、整改优先级和结论，不依赖外部 CDN。
- `findings.json`：供后续自动修复和复测读取。
- `evidence/`：脱敏后的扫描、SSH 和取证原始证据及哈希清单。

## 8. 风险控制与错误处理

第一版默认只读，允许读取状态、配置、日志、软件包、进程、端口和文件元数据。`sudo`、大范围日志读取、批量哈希和取证采集设置超时、并发及数据量限制。

禁止拒绝服务、暴力破解、持久化、后门、删除或篡改文件、导出真实敏感数据，以及明显改变业务状态的 PoC。超出安全 PoC 边界时停止并标记为需要人工验证。

优先使用 SSH Agent 或密钥文件。密码、私钥和 sudo 密码不得写入命令、日志、JSON 或报告。首次连接展示主机指纹，禁止静默忽略主机密钥变化。

网络失败只做有限重试和退避；SSH 中断后从未完成项目继续；权限不足、超时、工具缺失或输出损坏时记录 `not-assessable`。全面扫描使用保守并发与速率，发现高负载、服务异常或用户中止时立即停止。任何子流程失败仍需生成部分报告和覆盖率说明，不得把“未检查”表述为“安全”。

## 9. 测试与验收

测试覆盖：

- External、Internal、Both 与全面、定向范围的路由组合。
- Nmap XML、Nuclei JSONL、Xray JSON 和空结果样本。
- 越界阻止、报告定向验证、危险 PoC 拦截和敏感数据脱敏。
- 工具安装失败、网络超时、SSH 中断、sudo 不可用、磁盘不足和损坏输出。
- JSON Schema、证据引用、哈希、HTML 离线展示、响应式布局和打印效果。

集成测试分为两类：

1. 本地或隔离靶场：容器化 Linux SSH 目标、已知漏洞 Web 应用及预制取证样本。
2. 用户自有真实业务：允许执行实际安全验证和测试，但必须保持用户指定范围、非破坏 PoC、保守速率和异常即停；优先从定向检查开始，再按需要扩大到全面检查。真实业务的失败、超时和影响信号必须写入报告。

第一版验收标准：三种模式均能产生 Markdown、HTML、JSON 和证据目录；定向范围之外没有请求或文件访问；重复发现只计一次风险；子流程失败仍可交付可解释的部分结果；HTML 可离线交付领导查看；JSON 可被后续自动修复模块读取。

## 10. 非目标

第一版不自动修复服务器、不执行破坏性利用、不扩大目标范围、不将取证材料上传到第三方云服务，也不试图替代专业司法取证。自动修复将在后续阶段基于 `findings.json` 设计，并要求逐项确认、备份、回滚和复测。
